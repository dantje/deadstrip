1 /*******************************************************************************
2  * @mainpage
3  *
4  * @section faq Frequently Asked Questions
5  *
6  * <ol type="I">
7  * <li>What's all this about?</li>
8  * <ul type="none">
9  * <li>DeadStrip is a tool designed to perform dependency analysis and dead
10  * function/data removal on MinGW/gcc generated object files.</li>
11  * </ul>
12  *
13  * <li>But doesn't the compiler do this already?</li>
14  * <ul type="none">
15  * <li>No. The compiler doesn't know enough about the global contexts an
16  * object file might be linked into to do a reasonable strip of unreferenced
17  * code and data. It does however perform local optimization, i.e. in
18  * each object file.</li>
19  * </ul>
20  *
21  * <li>Okay, what about the linker?</li>
22  * <ul type="none">
23  * <li>Yes, the linker would be able to do dependency analysis, because it's
24  * got a global view onto the data. There's even a command line switch that
25  * enables this sort of global optimization; unfortunately it doesn't work
26  * under Windows.</li>
27  * </ul>
28  *
29  * <li>What is 'dead code stripping' anyway?</li>
30  * <ul type="none">
31  * <li>Removing code from your executable file that can be proven not to be
32  * in use. This reduces the total size of your apps.</li>
33  * </ul>
34  *
35  * <li>Couldn't I just remove all the unused functions by commenting them out?
36  * </li>
37  * <ul type="none">
38  * <li>You could, of course. But you're likely to forget some dependencies on
39  * larger projects, resulting in loads of linker errors and many hours of
40  * fun. Just imagine a larger framework you'd like to include in your
41  * project. You use just one function, because you're too lazy to reinvent
42  * the wheel. The compiler won't complain and the linker will link a nice XXL
43  * executable, not only including the PNG-loader but also a whole database
44  * management system, to manage your images in C++. This is the gap DeadStrip
45  * aims to fill.</li>
46  * </ul>
47  *
48  * <li>Then DeadStrip is an alternative linker?</li>
49  * <ul type="none">
50  * <li>No. You still use whatever linker you like to use. DeadStrip only
51  * hooks into the compiler-linker toolchain and analyses the object files
52  * before the linker sees them, removing all unused sections. When it's done,
53  * it calls the linker.</li>
54  * </ul>
55  *
56  * <li>Aren't there any other tools, that do the same already?</li>
57  * <ul type="none">
58  * <li>I've searched about one month for some tool like this, reading nearly
59  * every newsgroup entry containing the word GCC (there's a lot of them,
60  * trust me) and all I found, was some alternative linker contained in a
61  * large compiler-IDE framework that said it could do what I wanted. The
62  * final executable was larger than before when I was still linking using ld.
63  * Additionally ld has much more options when it comes down to fine tuning.
64  * So I decided to write that tool myself and still be able to use my
65  * favourite linker.</li>
66  * </ul>
67  *
68  * <li>I'm missing feature XY.</li>
69  * <ul type="none">
70  * <li>Then you've got a couple of options. If this feature is of major
71  * importance to you and you're not able to write it for yourself, ask me. If
72  * I like the idea, I might be implementing it and provide an update. \n
73  * But more reliable than me, is you. If you're able to write it for
74  * yourself, either by attaching another tool to the output of DeadStrip, or
75  * by contributing to the tools code, feel free to do so. DeadStrip provides
76  * XML-like output of various computed information and comes with full
77  * platform independent C source code (see below). I would be happy to hear
78  * from you in those cases. Although I strictly encourage you to contribute
79  * to the source, please don't provide links on your site to your modified
80  * version saying that this is the "official" one (that is, my one). You can,
81  * however, modify your own version and present it as a modification from
82  * you. Credits are not necessary.</li>
83  * </ul>
84  *
85  * <li>I found a bug.</li>
86  * <ul type="none">
87  * <li>Mail me with a detailed description of the error and provide some
88  * source code, if necessary, or try to fix it yourself and mail me
89  * afterwards :). My e-mail adress is weber &auml;t informatik.hu-berlin.de.
90  * </li>
91  * </ul>
92  *
93  * </ol>
94  *
95  * @section syntax Syntax
96  *
97  * @verbatim
98  deadstrip [Options] file...
99  Options:
100  --help display this help
101  --dcmd dumps the command line
102  --ddis dumps discarted sections
103  --duse dumps used sections
104  --dmap dump the dependency map
105  --linker <filename> use alternative linker (default: ld)
106  --dnrm do not remove any sections
107  --save <item> save an item and its dependencies
108  > just pass the decorated variable/function name, not the section
109  > the main function gets saved by default
110  @endverbatim
111  *
112  * @section notes Additional Notes
113  *
114  * <ul>
115  * <li>all dumps are printed to stdout - use the pipe symbol to redirect them,
116  * if necessary</li>
117  * <li>all switches that DeadStrip doesn't recognize are passed to the linker
118  * </li>
119  * <li>you should use the \c -ffunction-sections and/or \c -fdata-sections
120  * options, in order to generate appropriate section-partitioning; if you
121  * don't, DeadStrip won't be able to strip anything</li>
122  * <li>excessive section partitioning will increase the size of your exe, but
123  * compression tools like <a href="http://upx.sourceforge.net">UPX</a> can
124  * merge all sections afterwards</li>
125  * <li>when saving sections, be sure to only provide the decorated
126  * function/data name, not the prefix that the compiler generates</li>
127  * <ul>
128  * <li>plain C functions are not decorated at all, so when you need to save a
129  * function called "mainCRTStartup", pass this name to DeadStrip</li>
130  * <li>C++ functions are decorated according to their parameters, namespace
131  * and member information; you may need to look up their names using objdump
132  * from the binutils</li>
133  * <li>you can optionally place the functions that you want to be save into a
134  * section of your choice by using gcc's
135  * <tt>__attribute__((section("myName")))</tt>. DeadStrip doesn't strip code
136  * located in unknown sections.</li>
137  * </ul>
138  * <li>DeadStrip depends on some tools, that are provided by binutils</li>
139  * <ul>
140  * <li>namely objdump and objcopy</li>
141  * <li>that means, you should have DeadStrip located, where those two tools
142  * reside (e.g. MinGW/bin/)</li>
143  * </ul>
144  * </ul>
145  *
146  * @section install Installation
147  *
148  * Just drop the executable into the directory of your binutils, then configure
149  * your IDE or makefile to call DeadStrip instead of the linker, but provide the
150  * original linker using the --linker-switch. Finally enable
151  * \c -ffunction-sections and/or \c -fdata-sections in your compiler for your
152  * release builds.
